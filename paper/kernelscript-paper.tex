\title{\tool: Compiler-Checked Cross-Boundary Typed DSL for eBPF Applications}
% Replace TBD with the actual submission-system paper ID before submitting.
\author{Paper ID: 29}
\renewcommand{\shortauthors}{Paper ID: TBD}

\begin{abstract}
eBPF has become the standard mechanism for extending Linux with custom packet
processing, tracing, and security policies, but its programming model is
fragmented. A single application spans kernel code, a userspace loader, and
shared maps; the relationships among them (types, lifecycle ordering, shared
state) are checked only at load time or not at all, leading to late,
hard-to-diagnose failures. We observe that these cross-boundary relationships 
are duplicated concepts a type system can unify. We present \tool{}, a 
DSL that types maps, program handles, and
execution domains in one source, then lowers to standard C through the
clang/bpftool/libbpf toolchain. The compiler catches mismatches, such as wrong
context types or misordered lifecycle calls, before code generation. Generated
projects load, attach, and pass traffic on a real kernel, showing that eBPF's
cross-boundary structure can be typed while preserving the existing toolchain.
\end{abstract}

\section{Introduction}

% P1: Context/Importance
eBPF lets developers run application-specific logic inside the Linux kernel
under a verifier that proves memory safety and bounded
execution~\cite{linux-ebpf}. It now underpins packet processing in major cloud
providers, powers observability platforms, enforces security policies, and
enables scheduler experimentation. The ecosystem is large and growing.

% P2: Problem
Yet the programming model is fragmented. A single eBPF application spans kernel
code constrained by the verifier, a userspace loader that opens objects and
manages lifecycle, shared maps and ring buffers, and (for newer features)
generated skeleton headers that must match the running kernel. The relationships
among these pieces, such as a map's key and value types or the order of load,
attach, and detach, are checked only at load time or not at all. Errors surface
late as cryptic verifier rejections or silent mismatches.

% P3: Gap
Existing tools reduce boilerplate but leave cross-boundary relationships
implicit. The reference C/libbpf workflow requires hand-written kernel C,
userspace C, and a Makefile~\cite{libbpf}. Aya compiles both sides from Rust
and shares typed maps~\cite{aya,cilium-ebpf}, but lifecycle ordering remains a
runtime property. bpftrace raises the abstraction for tracing~\cite{bpftrace}
but does not target general eBPF applications. None of these tools type the
full cross-boundary structure at compile time.

% P4: Insight
\textbf{The key observation} is that these relationships are single concepts
duplicated across components. A map declaration is at once a kernel data
structure, a userspace lookup target, and a type contract; a program handle is
at once a function name, a BPF section, and a file descriptor. In C/libbpf,
naming conventions tie those copies together, so mismatches surface late. In
one typed source, the compiler can see and check them directly.

% P5: Solution
\tool{} realizes this idea as a compiled, multi-domain DSL. A developer writes
eBPF entry points, userspace control flow, maps, and ring buffers in one source
file. The compiler types maps, program handles, and execution domains, then
lowers to eBPF C, userspace C, and a Makefile. The ordinary libbpf and bpftool
path produces the final binaries, so the existing toolchain is preserved.

% P6: Contributions
This paper contributes:
\begin{itemize}
\item A \emph{typed model of cross-boundary eBPF structure}: execution domains,
  typed maps, and program-lifecycle handles, each with a typing rule and
  matching compiler diagnostic (Section~\ref{sec:model}).
\item A compiler that realizes this model while preserving the stock kernel
  toolchain, so compatibility failures remain visible (Section~\ref{sec:impl}).
\item An evaluation showing that each invariant fires, that one source generates
  buildable libbpf projects, and that generated objects load, attach, and run
  on a real kernel (Section~\ref{sec:eval}).
\end{itemize}

\section{Background and Motivation}
\label{sec:bg}

\textbf{eBPF application structure.} An eBPF application has at least two
components: a kernel program constrained by the verifier, and a userspace
loader that opens objects, attaches programs, and manages lifecycle. The two
sides communicate through maps, ring buffers, or perf events. For newer
features such as struct\_ops and kfuncs, bpftool generates skeleton headers
that the userspace code includes.

\textbf{Cross-boundary couplings.} This split creates three kinds of coupling.
\emph{Lifecycle coupling}: operations must occur in a valid order (load before
attach, detach before close), but C/libbpf checks this only at runtime.
\emph{Representation coupling}: a map's key and value types must agree across
kernel code, skeleton headers, and userspace lookups, but a mismatch is silent
until something fails. \emph{Compatibility coupling}: generated skeletons
depend on the exact kernel, bpftool, and libbpf versions, so they can disagree
with installed headers. The verifier catches many errors, but only at load
time. A DSL helps if it moves lifecycle and representation checks earlier,
into compile-time type errors.

\section{A Typed Model of Cross-Boundary Structure}
\label{sec:model}

This section sketches the parts of \tool's type system that encode the
couplings of Section~\ref{sec:bg}. We describe the model informally rather than
giving a full semantics or soundness proof; every rule corresponds to a
diagnostic exercised by the rejection suite of Section~\ref{sec:eval}.

A \tool{} program is a flat sequence of declarations. Listing~\ref{lst:min}
shows the minimal shape of an application; the rest of this section makes its
three relationships precise.

\begin{lstlisting}[language=C,caption={A unified-source eBPF application.},label={lst:min}]
include "xdp.kh"

@xdp fn pass_all(ctx: *xdp_md) -> xdp_action {
  return XDP_PASS
}

fn main() -> i32 {
  var prog = load(pass_all)
  attach(prog, "lo", 0)
  detach(prog)
  return 0
}
\end{lstlisting}

\textbf{Domains as attributes (representation coupling).} A function attribute
assigns each function to an execution domain
$d \in \{\textit{user}, \textit{ebpf}, \textit{helper}, \textit{kfunc}\}$.
\code{@xdp}, \code{@tc}, and \code{@probe} mark eBPF entry points. So do
\code{@tracepoint} and \code{@perf_event}. \code{@helper} marks kernel-shared
eBPF code; \code{@kfunc} and \code{@private} mark kernel-module code; an
unattributed \code{fn} is userspace. The checker tags each function with its
domain and rejects illegal crossings, for example a userspace function calling
an \code{@helper} or requesting kernel-context allocation from userspace.
Each eBPF attribute also fixes a context and return type: \code{@xdp} requires
\code{ctx:*xdp_md} returning \code{xdp_action}; \code{@tc} requires
\code{*__sk_buff} returning \code{i32}. The compiler rejects any signature that
disagrees with its attribute before code generation. The same surface syntax therefore remains
domain-aware: userspace can allocate and print freely, while eBPF code must
obey stack and helper restrictions. struct\_ops callbacks fit the same scheme,
with context and return types determined by the registered slot.

\textbf{Maps as typed globals (representation coupling).} A map is a typed
global, \code{var m : hash<K,V>(N)} or \code{array<K,V>(N)}, rather than a C
section plus separate userspace lookup code. The compiler checks element types
at every access site on \emph{both} sides of the kernel boundary: an index
\code{m[k]} requires \code{k:K} and yields \code{V}, in eBPF and userspace
alike. A wrong key type, wrong value type, or access to an
undeclared map is a compile-time error. One declaration thus replaces
kernel definitions, skeleton fields, and userspace lookups that otherwise must
agree by hand.

\textbf{Program lifecycle as typed handles (lifecycle coupling).} The most
protocol-like cross-boundary invariant is lifecycle. \tool{} distinguishes a
\code{FunctionRef} (a reference to an \code{@}-attributed function) from a
\code{ProgramHandle} (a loaded program), and types the lifecycle builtins so
that only \code{load} produces a \code{ProgramHandle}. The formation rule for
\code{FunctionRef} is also the rule that ties lifecycle to domains: only an
eBPF entry-point function inhabits \code{FunctionRef}, so an \code{@helper},
\code{@kfunc}, or userspace function is simply not a value \code{load} can
accept.
\begin{gather*}
\frac{f \text{ has an eBPF entry attribute}}
     {\Gamma \vdash f : \texttt{FunctionRef}}
\\[3pt]
\frac{\Gamma \vdash f : \texttt{FunctionRef}}
     {\Gamma \vdash \texttt{load}(f) : \texttt{ProgramHandle}}
\\[3pt]
\frac{\Gamma \vdash h : \texttt{ProgramHandle}}
     {\Gamma \vdash \texttt{detach}(h) : \texttt{unit}}
\\[3pt]
\frac{\Gamma \vdash h : \texttt{ProgramHandle}}
     {\Gamma \vdash \texttt{attach}(h, t, \mathit{fl}) : \texttt{u32}}
\end{gather*}
Because \code{attach} and \code{detach} consume a \code{ProgramHandle} and only
\code{load} produces one, ``attach before load'' is a \emph{type} error, not a
runtime failure. Passing a bare \code{FunctionRef}, an integer, or a string to
a lifecycle builtin does not type-check. This is a producer/consumer handle
discipline: because only \code{load} produces a \code{ProgramHandle},
it enforces the single ordering constraint
$\textit{load} \prec \{\textit{attach},\,\textit{detach}\}$ rather than the
complete load-to-attach-to-detach protocol. It guarantees that lifecycle
builtins receive values of the right kind, but not full path-sensitive
properties such as use-after-detach, double-attach, or leaked handles.

\textbf{From invariants to diagnostics.} The payoff is earlier, localized
compiler errors in place of naming conventions or late verifier/attach
failures. Section~\ref{sec:eval} measures whether those diagnostics fire;
Section~\ref{sec:limits} explains the limits of that evidence.

\section{Implementation}
\label{sec:impl}

The evaluated compiler is written in OCaml with dune. It parses \tool{} source,
resolves imports and includes, builds symbol tables, runs multi-program
analysis and type checking, performs safety analysis, builds an IR, and emits
C. For an input \texttt{foo.ks} the generated project normally contains
\texttt{foo.ebpf.c}, \texttt{foo.c}, a \texttt{Makefile}, and, when kfuncs are
present, \texttt{foo.mod.c}. The Makefile uses bpftool to dump
\texttt{vmlinux.h}, clang to produce a BPF object, bpftool to generate a
skeleton, and gcc to link the userspace loader against libbpf (together with
libelf and zlib).

\textbf{From typing rules to code.} The compiler realizes each invariant of
Section~\ref{sec:model} at a concrete point in this pipeline, which is why it
keeps \emph{separate modules} for maps, eBPF codegen, userspace codegen,
kernel-module generation, safety checking, BTF parsing, dynptr bridging,
tail-call analysis, and struct\_ops rather than acting as a thin preprocessor
over C. \emph{Domains} attach to each function as a scope refined by the entry
attribute; the IR records each function's program type, and code generation
dispatches the same source function to the eBPF or the userspace backend on
that field, so one surface syntax produces two ABIs without the author choosing
a target. \emph{Typed maps} lower to a single map descriptor that records the
key and value types once; both backends read that descriptor, so they emit the
eBPF \code{SEC(".maps")} definition and the userspace accessor from one source
of truth (Listing~\ref{lst:map}). The front end \emph{enforces lifecycle
handles}: only \code{load} yields a \code{ProgramHandle}, and \code{attach} and
\code{detach} require one, so a misordered call fails type checking before any
code generation. The handle then lowers to an ordinary file descriptor using standard libbpf
APIs.

\begin{lstlisting}[language=C,caption={One typed map declaration lowers to both
a kernel \texttt{SEC(".maps")} definition and a userspace accessor from a single
descriptor.},label={lst:map}]
// BPFScript source
var counts : hash<u32, u64>(1024)

// generated eBPF C
struct {
  __uint(type, BPF_MAP_TYPE_HASH);
  __uint(max_entries, 1024);
  __type(key, __u32);
  __type(value, __u64);
} counts SEC(".maps");

// generated userspace C: same key/value types
__u32 key = ...; __u64 val;
bpf_map_lookup_elem(counts_fd, &key, &val);
\end{lstlisting}

\textbf{Preserving the kernel toolchain.} The compiler does not replace the
verifier, clang, bpftool, or libbpf, so it inherits BTF, skeleton generation,
and existing diagnostics, and any compatibility failure remains visible and
measurable instead of being papered over. For interfaces that move faster than
the DSL can stabilize, \code{include} and \code{extern} declarations provide an
escape hatch to generated header declarations.

\section{Evaluation}
\label{sec:eval}

We evaluate \tool{} along three axes: \emph{(Q1)} does each typed
cross-boundary invariant fire as a compile-time diagnostic? \emph{(Q2)} does
one source generate a complete, buildable libbpf project? \emph{(Q3)} does
generated code load, attach, and run on a real kernel?

\textbf{Setup.} The evaluation runs on Linux \KSKernelVersion{} with clang 18,
bpftool 7.7, and libbpf 1.3.0. SLOC counts use nonblank, noncomment lines,
excluding generated \texttt{vmlinux.h} and skeleton headers. Timing numbers
are local samples reported as sanity checks rather than performance claims.

\subsection{Q1: The typed invariants fire}

\textbf{Regression coverage.} The compiler test suite reports
\KSTotalUnitSuites{} successful suites and \KSTotalUnitTests{} passing tests
with no failures, covering lexer/parser, type checking, maps and map
assignment, userspace codegen, context-field typing, safety, ring buffers,
perf-event attach, detach APIs, kfunc attributes, struct\_ops, and BTF
parsing, indicating that the implementation covers the domain concepts it
claims to encode rather than only a small example set.

\textbf{Static rejection suite.} The central Q1 result is a suite of
\KSStaticTotal{} targeted programs that exercises the diagnostics of
Section~\ref{sec:model} directly: one positive control plus
\KSStaticExpectedFailures{} expected rejections covering lifecycle misuses,
program-signature violations, map type errors, domain crossings, and safety
violations. Each case carries an expected diagnostic substring, so incidental
rejections do not count as passing. The suite shows that the invariants are implemented and stable.

\textbf{Differential failure-stage probe.} To ask whether the checks fire
\emph{earlier} than the stock toolchain on the same mistake, we construct
\KSOrganicTotal{} hand-chosen worked cases, each a realistic mistake injected
identically into a working XDP map-counter written in \tool{} and in matched
hand-written C/eBPF, and record the earliest stage each toolchain rejects it
along a shared ladder
(compile $<$ build $<$ verifier $<$ attach $<$ runtime $<$ undetected).
Table~\ref{tab:organic} reports the outcome. \tool{} rejects all
\KSOrganicTotal{} at compile time. The decisive case is the cross-boundary one:
an \texttt{@xdp} entry point handed a \texttt{*\_\_sk\_buff} context instead of
\texttt{*xdp\_md}. \tool{} rejects it from the program-signature rule of
Section~\ref{sec:model}. In C the program compiles, and because its body never
dereferences the context the verifier has no invalid access to flag, so the
mistake survives to a silently wrong load. The matched bodies deliberately touch
only a map, not a context field; one that did read a context field could instead
trip the verifier on an out-of-range access, but still would not be reported as
the context-type confusion it is. The
other \KSOrganicTie{} mistakes tie because clang already catches them locally:
an undeclared map, a string stored into a \texttt{u64} value, and an oversized
stack buffer. The result is therefore \KSOrganicKSEarlier{} earlier and
\KSOrganicTie{} tied: typing helps precisely on the cross-boundary relation that
local C checking cannot see. Section~\ref{sec:limits} explains the limits of
this small, author-injected set.

\subsection{Q2: One source generates the project}

\textbf{Buildability and centralization.} Nearly all examples compile and
build: \KSKSCompileOK{} of \KSTotalExamples{} compile, and \KSMakeOK{} produce
working libbpf projects. The single compile-time rejection is
\texttt{safety\_demo.ks}, where safety analysis detects 608 bytes of stack use
(above the 512-byte eBPF limit) and halts before codegen, a verifier-relevant
problem caught early. The two examples
that compile but do not build are both struct\_ops programs, and neither is a
compiler defect: each produces a valid eBPF object and fails only because the
bpftool-generated skeleton references a libbpf field absent from the host's
older installed headers, a toolchain version skew we isolate later in this
section (\emph{Where the toolchain leaks through}). For successful examples, the compiler emits a median of \KSMedianGeneratedSloc{}
lines from \KSMedianKSSloc{} source lines, a \KSMedianGeneratedRatio{} expansion
that measures emitted boilerplate rather than effort saved. A closer proxy is a
matched comparison: across \KSSourceFootprintWorkloads{} workloads with

Both corpora are author-written. Table~\ref{tab:features} shows

Most generated objects pass the verifier: \KSVerifierBuildLoadOK{} of

\textbf{Runtime sanity.} Beyond load and attach, matched workloads confirm the
generated objects \emph{run}. Under \code{BPF_PROG_TEST_RUN}, a minimal
generated XDP pass program matches its hand-written baseline in instruction
count and latency. Over \KSTrafficTrials{} veth/iperf3 trials, generated and
hand-written XDP pass programs deliver comparable throughput, and matched TC
programs pass traffic as well.

\textbf{Where the toolchain leaks through.} Preserving the kernel toolchain
means \tool{} also inherits its compatibility coupling. The two struct\_ops
build failures are not parser or type errors: both compile to valid eBPF
objects but fail because the bpftool-generated skeleton references a libbpf
field absent from the installed headers. This is a mismatched-toolchain pairing,
not an inherent limitation. A direct libbpf runner (bypassing skeletons) loads
both the generated and a matched hand-written tcp-congestion struct\_ops object
successfully. The failure is a version-skew problem in the generated
\emph{skeleton}, exactly the compatibility coupling of Section~\ref{sec:bg}
that source unification does not eliminate.

\section{Limitations}
\label{sec:limits}

The differential probe injects only \KSOrganicTotal{} hand-chosen mistakes;
a stronger study would mine real eBPF bugs from revision histories.
Lifecycle typing enforces a producer/consumer handle distinction but not
full typestate, so use-after-detach and leaked handles are not statically
ruled out. The corpus is author-written; the external port shows the
line-count advantage does not transfer unchanged. Traffic numbers are local
sanity checks, not performance claims. Finally, the struct\_ops build
failures reflect bpftool/libbpf version skew rather than a deep limitation,
but they show that generated builds depend on toolchain alignment.

\section{Related Work}

\textbf{C/libbpf and host-language bindings.} XDP established the programmable
in-kernel packet model \tool{} targets~\cite{xdp}; surveys cover the broader
eBPF landscape~\cite{ebpfsurvey,linux-ebpf}. C/libbpf is the reference workflow
but leaves kernel code, loaders, skeletons, maps, and build logic in separate
files~\cite{libbpf}. Scaffolding such as libbpf-bootstrap and xdp-tutorial
cuts boilerplate yet leaves the boundary relationships
untyped~\cite{libbpf-bootstrap,xdp-tutorial}. Host-language libraries go part
of the way. Aya compiles both sides from one Rust source and shares typed maps
across the boundary~\cite{aya,cilium-ebpf}.
\tool{} differs in being host-language-independent: execution domains and
crossing rules live in the DSL itself, and it lowers to the standard Linux BPF
toolchain rather than a host runtime, so BTF, CO-RE, skeletons, and verifier
diagnostics are inherited and compatibility skew stays measurable. Rex similarly narrows the
language-verifier gap with safe Rust extensions, but like Aya targets a host
language and its own loading path rather than stock libbpf~\cite{rex}.

\textbf{Tracing and bytecode tools.} bpftrace is concise but narrows the target
to tracing~\cite{bpftrace}. BCC scripts eBPF from Python but keeps Python as a
controller around separate BPF code~\cite{bcc}; related systems use eBPF to
\emph{observe} distributed LLM inference~\cite{einfer-ebpf} or move the runtime
itself to userspace~\cite{bpftime}. A separate line optimizes the eBPF bytecode
after it exists: PREVAIL~\cite{prevail}, K2~\cite{k2}, and
hXDP~\cite{hxdp}; \tool{} is earlier and complementary, generating the
bytecode and its loader.

\textbf{Dataplane DSLs and typestate.} Compiling one typed description into a
split low-level implementation has precedent: P4 compiles a
protocol-independent program into target parser and match-action
pipelines~\cite{p4}, and Click composes packet processing from typed
elements~\cite{click}. Unlike a P4 pipeline, an eBPF application must also
generate its userspace coordinator and survive a late general-purpose verifier
and fast-moving libbpf/bpftool ABIs, the source of the compatibility limits we
measure. \tool's lifecycle discipline is a lightweight instance of
typestate~\cite{typestate}: we enforce only the producer/consumer handle
distinction, so the connection is a design influence rather than a claim of
equivalent guarantees. The unrealized guarantees are substructural: an affine
\texttt{ProgramHandle} consumed by \texttt{detach} (or a typed \texttt{Link}
returned by \texttt{attach}) would rule out use-after-detach, double-attach,
and leaks~\cite{lineartypes}. Vault enforces such resource protocols in
low-level software~\cite{vault}, and session types formalize the ordering more
fully~\cite{sessiontypes}. Our domain separation echoes address-space type
qualifiers such as the kernel's Sparse \texttt{\_\_user}/\texttt{\_\_kernel}
annotations~\cite{sparse}. \tool{} thus occupies a point not covered above: a
compiled, multi-domain, single-source language whose question is whether eBPF
application structure can be made explicit enough for a compiler to type and
generate the low-level project \emph{while preserving} the existing Linux BPF
toolchain.

\section{Conclusion}

\tool{} treats the cross-boundary structure of an eBPF application (program
lifecycle, shared maps, and execution domains) as something a compiler can type
rather than something developers hold together by convention and discover at
verifier-load or attach time. The evaluation confirms that each typed invariant
fires as a compile-time diagnostic, that one source generates buildable libbpf
projects, and that generated objects load, attach, and pass traffic on a real
kernel. The result is a concrete design point: typing eBPF's cross-boundary
structure is feasible, and because the compiler lowers through the stock
toolchain, both what the type system buys and the version-skew limits it cannot
absorb stay visible and measurable.
