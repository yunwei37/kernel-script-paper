\title{\tool: Compiler-Checked Cross-Boundary Types for eBPF Applications}

\begin{abstract}
A realistic eBPF application spans several coupled artifacts: a
verifier-constrained kernel program, a userspace loader, shared maps, and
generated skeletons. The relationships across them (which program a handle
refers to, a map's key and value type on both sides of the kernel boundary, and
the load-to-attach-to-detach order) are usually implicit naming conventions
checked only at verifier-load or attach time. We study \tool, a typed DSL that
lifts these relationships into the type system: typed maps, typed
program-lifecycle handles, and execution domains turn cross-boundary
mismatches into compiler diagnostics while still lowering to ordinary C through
the stock clang/bpftool/libbpf toolchain. On the artifact at commit
\KSCommitShort, a suite of \KSStaticExpectedFailures{} invalid programs
confirms that each invariant fires; \KSKSCompileOK{} of \KSTotalExamples{}
examples compile and \KSMakeOK{} build into libbpf-compatible projects; and
the generated objects verifier-load, attach, pass traffic, and run a
tcp-congestion struct\_ops object on a real kernel through a direct libbpf
runner. The result is a concrete design point for typed cross-boundary eBPF
structure that preserves the kernel toolchain while exposing its remaining
compatibility boundaries.
\end{abstract}

\section{Introduction}

eBPF lets Linux developers run application-specific logic inside the kernel
under a verifier that proves memory safety and bounded
execution~\cite{linux-ebpf}. It now underpins packet processing, traffic
control, tracing, security policy, and scheduler experimentation. Yet the
programming model is awkward for a structural reason: a single eBPF
\emph{application} is split across artifacts that must agree with one another.
The kernel program is written under verifier constraints; a userspace
coordinator opens objects, finds maps, installs links, and tears resources
down; maps and ring buffers carry shared state; BTF describes kernel types; and
newer interfaces such as kfuncs and struct\_ops require generated skeleton code
to match the installed kernel headers.

The reference C/libbpf workflow is powerful but verbose \cite{libbpf}: the
developer hand-writes kernel C, userspace C, map setup, attachment and pinning
logic, cleanup paths, and a Makefile. Libraries in Rust, Go, or Python improve
this experience, and some (notably Aya) author both sides from one host-language
source~\cite{aya,cilium-ebpf}; tracing languages such as
bpftrace raise the abstraction level for one
domain~\cite{bpftrace}. But even when both sides are authored together, the
relationships that span the kernel boundary, lifecycle ordering and domain
separation, stay largely runtime or convention rather than compiler-checked
types.

\textbf{The key observation} is that these relationships are single concepts
duplicated across artifacts. A map declaration is at once a kernel data
structure, a userspace lookup target, and a type contract; a program reference
is at once a function name, BPF section, skeleton field, and file descriptor.
In C/libbpf, those copies are tied together by naming conventions and generated
headers, so mismatches surface late. In one typed source, the compiler can see
and check them directly.

This paper studies that idea through \tool, a compiled, multi-domain DSL in
which a developer writes eBPF entry points, helpers, userspace control flow,
maps, ring buffers, perf events, and optional kernel-module code in one source
language. The compiler lowers this source to eBPF C, userspace C, an optional
module, and a Makefile, after which the ordinary libbpf and bpftool path
produces objects, skeletons, and binaries. \tool{} is beta software; its value
here is as a prototype for studying the design space, not as a libbpf
replacement.

This paper makes three contributions. \emph{(1)}~We formulate \tool's central
idea as a \emph{typed model of cross-boundary eBPF structure}: execution
domains, typed maps shared across the kernel boundary, and a program lifecycle
modeled as typed producer/consumer handles, each specified as a typing rule
with a matching compiler diagnostic (Section~\ref{sec:model}). \emph{(2)}~We
show how the prototype realizes that model while preserving the stock kernel
toolchain, so compatibility failures remain visible and measurable rather than
hidden (Section~\ref{sec:impl}). \emph{(3)}~We give a reproducible artifact
evaluation (Section~\ref{sec:eval}) that confirms each typed invariant fires,
that one source generates buildable libbpf projects, and that the generated
objects load, attach, and run on a real kernel, while being explicit about the
limits of that evidence. All scripts, logs, derived tables, and paper macros
live outside the cloned \tool{} tree so the evaluation can be rerun without
editing the source under test.

\section{Background: Three Couplings}
\label{sec:bg}

An eBPF application has at least two components, a restricted kernel program
and a userspace coordinator, communicating through maps, ring buffers, perf
events, or global data. This split creates three kinds of coupling that a
language could, in principle, check.

\textbf{Lifecycle coupling.} Operations must occur in a valid order: an object
is opened, loaded, attached, read, and detached using type-appropriate APIs.
``Attach before load'' or ``read a handle that was never produced'' are
ordering errors that C/libbpf catches only at runtime.

\textbf{Representation coupling.} The same concept (a map, a typed context
field) must be expressed consistently in kernel C, generated skeleton headers,
userspace lookup code, and BTF-derived declarations. A key or value type that
disagrees across those copies is a silent mismatch until something fails.

\textbf{Compatibility coupling.} eBPF interfaces evolve quickly. struct\_ops,
kfuncs, dynptrs, and scheduler extensions depend on the exact kernel, bpftool,
and libbpf versions, so generated skeletons can disagree with installed
headers.

The verifier is essential but deliberately late. A developer may write,
compile, generate skeletons, link a loader, and only then receive a rejection.
A DSL helps if it moves the first two couplings earlier, into type errors,
without hiding the execution model or abandoning the toolchain that enforces
the third.

\section{A Typed Model of Cross-Boundary Structure}
\label{sec:model}

This section sketches the parts of \tool's type system that encode the
couplings of Section~\ref{sec:bg}. We describe the model informally rather than
giving a full semantics or soundness proof. Every rule corresponds to a
diagnostic exercised by the rejection suite of Section~\ref{sec:eval}
(parenthesized names are the cases in \texttt{experiments/static\_checks/}).

A \tool{} program is a flat sequence of declarations. Listing~\ref{lst:min}
shows the minimal shape of an application; the rest of this section makes its
three relationships precise.

\begin{lstlisting}[language=C,caption={A unified-source eBPF application.},label={lst:min}]
include "xdp.kh"

@xdp fn pass_all(ctx: *xdp_md) -> xdp_action {
  return XDP_PASS
}

fn main() -> i32 {
  var prog = load(pass_all)
  attach(prog, "lo", 0)
  detach(prog)
  return 0
}
\end{lstlisting}

\textbf{Domains as attributes (representation coupling).} A function attribute
assigns each function to an execution domain
$d \in \{\textit{user}, \textit{ebpf}, \textit{helper}, \textit{kfunc}\}$.
\code{@xdp}, \code{@tc}, and \code{@probe} mark eBPF entry points. So do
\code{@tracepoint} and \code{@perf_event}. \code{@helper} marks kernel-shared
eBPF code; \code{@kfunc} and \code{@private} mark kernel-module code; an
unattributed \code{fn} is userspace. The checker tags each function with its
domain and rejects illegal crossings, for example a userspace function calling
an \code{@helper} (\code{helper_from_userspace}) or requesting kernel-context
allocation from a userspace or XDP domain
(\code{gfp_from_userspace}, \code{gfp_from_xdp}). Each eBPF attribute also fixes
a context and return type: \code{@xdp} requires \code{ctx:*xdp_md} returning
\code{xdp_action}; \code{@tc} requires \code{*__sk_buff} returning \code{i32}. A
signature that disagrees with its attribute is rejected before code generation
(\code{xdp_wrong_context}, \code{tc_wrong_context}, \code{xdp_wrong_return},
\code{probe_wrong_return}). The same surface syntax therefore remains
domain-aware: userspace can allocate and print freely, while eBPF code must
obey stack and helper restrictions. struct\_ops callbacks fit the same scheme,
with context and return types determined by the registered slot.

\textbf{Maps as typed globals (representation coupling).} A map is a typed
global, \code{var m : hash<K,V>(N)} or \code{array<K,V>(N)}, rather than a C
section plus separate userspace lookup code. Its element types form one
contract checked at every access site on \emph{both} sides of the kernel
boundary: an index \code{m[k]} requires \code{k:K} and yields \code{V}, in eBPF
and userspace alike. A wrong key type, wrong value type, or access to an
undeclared map is rejected (\code{map_string_key},
\code{map_string_value}, \code{map_undefined}). One declaration thus replaces
kernel definitions, skeleton fields, and userspace lookups that otherwise must
agree by hand.

\textbf{Program lifecycle as typed handles (lifecycle coupling).} The most
protocol-like cross-boundary invariant is lifecycle. \tool{} distinguishes a
\code{FunctionRef} (a reference to an \code{@}-attributed function) from a
\code{ProgramHandle} (a loaded program), and types the lifecycle builtins so
that only \code{load} produces a \code{ProgramHandle}. The formation rule for
\code{FunctionRef} is also the rule that ties lifecycle to domains: only an
eBPF entry-point function inhabits \code{FunctionRef}, so an \code{@helper},
\code{@kfunc}, or userspace function is simply not a value \code{load} can
accept.
\begin{gather*}
\frac{f \text{ has an eBPF entry attribute}}
     {\Gamma \vdash f : \texttt{FunctionRef}}
\\[3pt]
\frac{\Gamma \vdash f : \texttt{FunctionRef}}
     {\Gamma \vdash \texttt{load}(f) : \texttt{ProgramHandle}}
\\[3pt]
\frac{\Gamma \vdash h : \texttt{ProgramHandle}}
     {\Gamma \vdash \texttt{detach}(h) : \texttt{unit}}
\\[3pt]
\frac{\Gamma \vdash h : \texttt{ProgramHandle}}
     {\Gamma \vdash \texttt{attach}(h, t, \mathit{fl}) : \texttt{u32}}
\end{gather*}
Because \code{attach} and \code{detach} consume a \code{ProgramHandle} and only
\code{load} produces one, ``attach before load'' is a \emph{type} error, not a
runtime failure. Passing a bare \code{FunctionRef}
(\code{attach_without_load}), an integer (\code{attach_integer_handle}), or a
string (\code{load_string}, \code{detach_string}) to a lifecycle builtin does
not type-check. This is a producer/consumer handle discipline, not yet a full
typestate automaton: because only \code{load} produces a \code{ProgramHandle},
it enforces the single ordering constraint
$\textit{load} \prec \{\textit{attach},\,\textit{detach}\}$ rather than the
complete load-to-attach-to-detach protocol. It guarantees that lifecycle
builtins receive values of the right kind, but not full path-sensitive
properties such as use-after-detach, double-attach, or leaked handles.

\textbf{From invariants to diagnostics.} The payoff is earlier, localized
compiler errors in place of naming conventions or late verifier/attach
failures. Section~\ref{sec:eval} measures whether those diagnostics fire;
Section~\ref{sec:limits} explains the limits of that evidence.

\section{Implementation}
\label{sec:impl}

The evaluated compiler is written in OCaml with dune. It parses \tool{} source,
resolves imports and includes, builds symbol tables, runs multi-program
analysis and type checking, performs safety analysis, builds an IR, and emits
C. For an input \texttt{foo.ks} the generated project normally contains
\texttt{foo.ebpf.c}, \texttt{foo.c}, a \texttt{Makefile}, and, when kfuncs are
present, \texttt{foo.mod.c}. The Makefile uses bpftool to dump
\texttt{vmlinux.h}, clang to produce a BPF object, bpftool to generate a
skeleton, and gcc to link the userspace loader against libbpf (together with
libelf and zlib).

\textbf{From typing rules to code.} Each invariant of
Section~\ref{sec:model} is realized at a concrete point in this pipeline, which
is why the compiler keeps \emph{separate modules} for maps, eBPF codegen,
userspace codegen, kernel-module generation, safety checking, BTF parsing,
dynptr bridging, tail-call analysis, and struct\_ops rather than acting as a
thin preprocessor over C: encoding eBPF concepts directly is what gives each
check and each lowering somewhere to live. \emph{Domains} are carried as a
per-function scope refined by the entry attribute; the IR records each
function's program type, and code generation dispatches the same source function
to the eBPF or the userspace backend on that field, so one surface syntax
produces two ABIs without the author choosing a target. \emph{Typed maps} lower
to a single map descriptor that records the key and value types once; both
backends read that descriptor, so the eBPF \code{SEC(".maps")} definition and
the userspace accessor are emitted from one source of truth
(Listing~\ref{lst:map}). \emph{Lifecycle handles} are enforced in the front
end: only \code{load} is typed to yield a \code{ProgramHandle}, and the
\code{attach} and \code{detach} builtins are typed to require one, so a
misordered call fails type checking before any code is emitted. The handle then
lowers to an ordinary file descriptor: \code{load} resolves the program by name
(\code{bpf_object__find_program_by_name}) and takes its descriptor
(\code{bpf_program__fd}), \code{attach} dispatches on the program type to the
matching libbpf entry point (\code{bpf_xdp_attach},
\code{bpf_program__attach_tcx}, \code{bpf_program__attach_kprobe}, and so on),
and \code{detach} releases the link through \code{bpf_link__destroy}. The typed
discipline thus lives entirely in the compiler and disappears into stock libbpf
calls.

\begin{lstlisting}[language=C,caption={One typed map declaration lowers to both
a kernel \texttt{SEC(".maps")} definition and a userspace accessor, emitted from
a single descriptor so the key and value types cannot drift apart.},label={lst:map}]
// KernelScript source
var counts : hash<u32, u64>(1024)

// generated eBPF C
struct {
  __uint(type, BPF_MAP_TYPE_HASH);
  __uint(max_entries, 1024);
  __type(key, __u32);
  __type(value, __u64);
} counts SEC(".maps");

// generated userspace C: same key/value types
__u32 key = ...; __u64 val;
bpf_map_lookup_elem(counts_fd, &key, &val);
\end{lstlisting}

\textbf{Preserving the kernel toolchain.} The compiler does not replace the
verifier, clang, bpftool, or libbpf, so it inherits BTF, skeleton generation,
and existing diagnostics, and any compatibility failure remains visible and
measurable instead of being papered over. For interfaces that move faster than
the DSL can stabilize, \code{include} and \code{extern} declarations provide an
escape hatch to generated header declarations.

\textbf{Two further lowerings} show the model reaches beyond the immediate
kernel/userspace split, though neither is central to the typed-boundary claim.
Source-level ring-buffer operations (\code{reserve}, field write,
\code{submit}) lower to the appropriate dynptr helpers, so the author never
hand-assembles that lifetime. A userspace stage may also \code{exec()} into
Python and reuse open map descriptors: the loader clears \code{FD_CLOEXEC},
passes descriptor numbers through the environment, and the compiler emits a
Python wrapper that reopens typed access from compile-time metadata
(\code{test_exec.ks}). The map schema and descriptor handoff still come from one
typed source rather than hand-written glue.

\section{Evaluation}
\label{sec:eval}

We evaluate \tool{} as a compiler and systems artifact, asking three questions
that track the contributions: \emph{(Q1)} does each typed cross-boundary
invariant actually fire as a compile-time diagnostic? \emph{(Q2)} does one
typed source generate a complete, buildable libbpf project? \emph{(Q3)} are the
generated artifacts real, that is, do they load, attach, and run on a kernel
rather than merely compile?

\textbf{Setup.} The evaluation runs on Linux \KSKernelVersion{} with clang 18,
bpftool 7.7, libbpf 1.3.0, OCaml 4.14, and dune 3.14, on a \KSCPUModel{} with
\KSCPUCount{} CPUs, at commit \KSCommitShort. The harness writes raw stdout and
stderr for every compiler and make invocation, per-example CSV rows,
per-script JSON, and the numeric macros used in this paper, so every aggregate
traces back to a logged row rather than a hand-edited table. SLOC counts use
nonblank, noncomment lines. We exclude generated \texttt{vmlinux.h} and
skeleton headers from expansion factors because bpftool emits them, not
\tool{}. Timing numbers below are local, unpinned samples (CPU
governor \KSCPUGovernor, turbo \KSTurboState, no CPU pinning, virtualization
\KSVirtualization) reported as sanity evidence rather than performance claims;
we report medians with ranges.

\subsection{Q1: The typed invariants fire}

\textbf{Regression coverage.} The compiler test suite reports
\KSTotalUnitSuites{} successful suites and \KSTotalUnitTests{} passing tests
with no failures, covering lexer/parser, type checking, maps and map
assignment, userspace codegen, context-field typing, safety, ring buffers,
perf-event attach, detach APIs, kfunc attributes, struct\_ops, and BTF
parsing, indicating that the implementation covers the domain concepts it
claims to encode rather than only a small example set.

\textbf{Static rejection suite.} The central Q1 result is a suite that
exercises the diagnostics of Section~\ref{sec:model} directly. The harness
compiles \KSStaticTotal{} targeted programs and verifies that every outcome
matches expectation: one positive control plus \KSStaticExpectedFailures{}
expected rejections: \KSStaticLifecycle{} lifecycle API misuses,
\KSStaticProgramSignature{} program-signature violations, \KSStaticMapType{}
map type/symbol errors, \KSStaticTypeSystem{} general type errors,
\KSStaticSymbolValidation{} symbol-validation errors,
\KSStaticConfigBoundary{} config-boundary violation,
\KSStaticHelperScope{} helper-scope violation,
\KSStaticKernelContext{} kernel-context allocation violations,
\KSStaticPerfEventGroup{} perf-event group-bound violations,
\KSStaticRingbuf{} ring-buffer API misuse, and \KSStaticSafety{} stack-limit
violation. These cases line up one-to-one with the lifecycle, map, and domain
rules of Section~\ref{sec:model}. The oracle is stronger than a nonzero exit
code: each case carries an expected diagnostic substring, so incidental
rejections do not count as passing. The suite is therefore a
\emph{diagnostic-contract regression test}: it shows that the invariants are
implemented, mapped to the intended error classes, and stable across compiler
changes. It does not by itself show how often those checks catch real mistakes;
the next paragraph probes that question.

\textbf{Differential failure-stage probe.} To ask whether the checks fire
\emph{earlier} than the stock toolchain on the same mistake, we pre-register
\KSOrganicTotal{} realistic mistakes, each injected identically into a working
XDP map-counter written in \tool{} and in matched hand-written C/eBPF, and
record the earliest stage each toolchain rejects it along a shared ladder
(compile $<$ build $<$ verifier $<$ attach $<$ runtime $<$ undetected).
Table~\ref{tab:organic} reports the outcome with the kernel verifier live
(verifier tested: \KSOrganicVerifierTested). \tool{} rejects all
\KSOrganicTotal{} at compile time. The decisive case is the cross-boundary one:
an \texttt{@xdp} entry point handed a \texttt{*\_\_sk\_buff} context instead of
\texttt{*xdp\_md}. \tool{} rejects it from the program-signature rule of
Section~\ref{sec:model}, whereas clang compiles it and the verifier
\emph{accepts} it, so in C the mistake survives to a silently wrong load. The
other \KSOrganicTie{} mistakes tie because clang already catches them locally:
an undeclared map, a string stored into a \texttt{u64} value, and an oversized
stack buffer. The result is therefore \KSOrganicKSEarlier{} earlier and

local C checking cannot see. Section~\ref{sec:limits} explains the limits of
this small, author-injected set.

\subsection{Q2: One source generates the project}

\KSKSCompileOK{} compile from \tool{} source (\KSKSCompilePct) and \KSMakeOK{}

script then takes the verifier-clean single-section XDP subset, attaches each
object to a fresh veth in its own network namespace, confirms
\code{prog/xdp} in \code{ip -d link show}, and detaches it. \KSAttachOK{} of
\KSAttachEligible{} attach and detach cleanly (\KSAttachOKPct). The attach
matrix covers XDP; TC and struct\_ops are exercised by the workloads below,
while probes, tracepoints, perf events, and ring buffers are verifier-loaded
but not attach-tested here. The ``runs on a real kernel'' claim is therefore
scoped to XDP, TC, and struct\_ops.

\textbf{Runtime sanity.} Beyond load and attach, matched workloads confirm the
generated objects \emph{run}. Under \code{BPF_PROG_TEST_RUN}, a minimal
generated XDP pass program matches its hand-written baseline in both
instruction count and median latency (\KSMicroKsPassInstr{} instructions,
\KSMicroKsPassNs{} vs.\ \KSMicroCPassNs{}). A map-counter program does not: the
generated object emits \KSMicroKsCountInstr{} instructions versus
\KSMicroCCountInstr{} for C (\KSMicroKsCountNs{} vs.\ \KSMicroCCountNs{}). That
comparison is not matched because the C baseline uses an atomic add while the
generated path lowers to a separate lookup and update. This gap reflects a
backend lowering choice rather than the typed model itself. Over
\KSTrafficTrials{} veth/iperf3 trials, generated and hand-written XDP pass
programs deliver comparable local medians
(\KSTrafficKsPassGbps{} vs.\ \KSTrafficCPassGbps{} Gb/s), and matched TC
programs pass traffic as well. These are local, unpinned veth/TCP samples that
show the generated object runs without collapsing the data path.

\textbf{Where the toolchain leaks through.} Preserving the kernel toolchain
means \tool{} also inherits its compatibility coupling, and the artifact makes
this visible rather than hiding it. The two generated build failures
(\code{sched_ext_simple.ks}, \code{struct_ops_simple.ks}) are not parser or type
errors: both compile to eBPF objects and generate skeletons. But bpftool
\KSStructOpsBpftoolVersion{} emits a generated skeleton that assigns the field
\code{bpf_map_skeleton.link}; the older installed libbpf
\KSStructOpsLibbpfVersion{} headers on our host do not provide that field
(\KSStructOpsMismatchPct{} of examples). This is a mismatched-toolchain pairing,
with newer bpftool against older libbpf, not an inherent property of either
version. A matched bpftool/libbpf pair would not exhibit it. To separate this
skeleton-header skew from object compatibility, a shared libbpf runner loads
both the generated and a matched hand-written tcp-congestion struct\_ops object
\emph{without} generated skeletons. On this host (bpftool
\KSStructOpsBpftoolVersion, libbpf \KSStructOpsLibbpfVersion, skeleton map-link
support \KSStructOpsSkeletonLinkSupported) both objects load, attach, and
detach in \KSStructOpsKsDetachOK/\KSStructOpsTrials{} and
\KSStructOpsCDetachOK/\KSStructOpsTrials{} trials. The generated object is
therefore valid; the failure is a version-skew problem in the generated
\emph{skeleton}, exactly the compatibility coupling of Section~\ref{sec:bg}
that source unification does not eliminate.

\section{Limitations and Discussion}
\label{sec:limits}

The evaluation supports a bounded reading of the contribution and clarifies
what remains open.

\textbf{The organic probe is small.} Section~\ref{sec:eval} injects only
\KSOrganicTotal{} realistic mistakes into matched \tool{} and C/eBPF examples,
finding \tool{} strictly earlier on \KSOrganicKSEarlier{} and tied on
\KSOrganicTie. That establishes a preview, not a population estimate. A
stronger study would mine real eBPF mistakes from revision histories and issue
trackers, port them to \tool, and report the resulting failure stages.

\textbf{Lifecycle typing remains lightweight.} It enforces the
producer/consumer handle distinction, not full flow-sensitive typestate, so
use-after-detach, double-attach, and leaked handles are not statically ruled
out on every path.

\textbf{The corpus is author-written.} The expansion and feature results come
from the \tool{} repository, which reflects features the authors chose to
demonstrate; the external port shows the line-count advantage does not transfer
unchanged. A stronger study would reimplement programs from Cilium,
libbpf-tools, and production filters in both \tool{} and C/libbpf.

\textbf{No runtime-equivalence claim.} Traffic and microbenchmark numbers are
local, unpinned veth/TCP samples that establish sanity, not performance. A
deployment study would run each example in a VM with a known kernel
configuration, representative workloads, and explicit post-detach cleanup
checks.

\textbf{Toolchain version alignment matters.} The struct\_ops result is a
mismatched bpftool/libbpf pairing rather than a deep limitation, but it shows
that a generated build depends on alignment between kernel headers, bpftool,
generated skeletons, and libbpf. A production \tool{} should record the target
kernel ABI, check libbpf feature support before generation, and fail early with
a dependency diagnostic.

\textbf{The Python bridge is a mechanism, not yet a result.} The
\texttt{exec()}-to-Python path lets the compiler, not the developer, carry
typed maps across the process boundary, but the checked-in example stops at
direct map reads and writes: it is a usable path into the Python ecosystem, not
a machine-learning integration result.

\section{Related Work}

\textbf{C/libbpf and host-language bindings.} XDP established the programmable
in-kernel packet model \tool{} targets~\cite{xdp}; surveys cover the broader
eBPF landscape~\cite{ebpfsurvey,linux-ebpf}. C/libbpf is the reference workflow
but leaves kernel code, loaders, skeletons, maps, and build logic as separate
artifacts~\cite{libbpf}. Scaffolding such as libbpf-bootstrap and xdp-tutorial
cuts boilerplate yet leaves the boundary relationships
untyped~\cite{libbpf-bootstrap,xdp-tutorial}. Host-language libraries go part
of the way: Aya in particular compiles both sides from one Rust source and
shares typed maps across the boundary~\cite{aya,cilium-ebpf}, so single-source
authoring and typed maps are not by themselves \tool's contribution. \tool{}
instead is host-language-independent, with execution domains and crossing rules
in the language. It lowers to the standard Linux BPF toolchain rather than a
host runtime, so BTF, CO-RE, skeletons, and verifier diagnostics are inherited
and compatibility skew stays measurable. Rex similarly narrows the
language-verifier gap with safe Rust extensions, but like Aya targets a host
language and its own loading path rather than stock libbpf~\cite{rex}.

\textbf{Tracing and bytecode tools.} bpftrace is concise but narrows the target
to tracing~\cite{bpftrace}. BCC scripts eBPF from Python but keeps Python as a
controller around separate BPF code~\cite{bcc}; related systems use eBPF to
\emph{observe} distributed LLM inference~\cite{einfer-ebpf} or move the runtime
itself to userspace~\cite{bpftime}. A separate line optimizes the eBPF bytecode
after it exists: PREVAIL~\cite{prevail}, K2~\cite{k2}, and
hXDP~\cite{hxdp}; \tool{} is earlier and complementary, generating the
bytecode and its loader.

\textbf{Dataplane DSLs and typestate.} Compiling one typed description into a
split low-level implementation has precedent: P4 compiles a
protocol-independent program into target parser and match-action
pipelines~\cite{p4}, and Click composes packet processing from typed
elements~\cite{click}. Unlike a P4 pipeline, an eBPF application must also
generate its userspace coordinator and survive a late general-purpose verifier
and fast-moving libbpf/bpftool ABIs, the source of the compatibility limits we
measure. \tool's lifecycle discipline is a lightweight instance of
typestate~\cite{typestate}: we enforce only the producer/consumer handle
distinction, so the connection is a design influence rather than a claim of
equivalent guarantees. The unrealized guarantees are substructural: an affine
\texttt{ProgramHandle} consumed by \texttt{detach} (or a typed \texttt{Link}
returned by \texttt{attach}) would rule out use-after-detach, double-attach,
and leaks~\cite{lineartypes}, Vault enforces such resource protocols in
low-level software~\cite{vault}, and session types formalize the ordering more
fully~\cite{sessiontypes}. Our domain separation echoes address-space type
qualifiers such as the kernel's Sparse \texttt{\_\_user}/\texttt{\_\_kernel}
annotations~\cite{sparse}. \tool{} thus occupies a point not covered above: a
compiled, multi-domain, single-source language whose question is whether eBPF
application structure can be made explicit enough for a compiler to type and
generate the low-level project \emph{while preserving} the existing Linux BPF
toolchain.

\section{Conclusion}

\tool{} treats the cross-boundary structure of an eBPF application (program
lifecycle, shared maps, and execution domains) as something a compiler can type
rather than something developers hold together by convention and discover at
verifier-load or attach time. On the checked-in artifact, a suite of
\KSStaticExpectedFailures{} invalid programs confirms each typed invariant fires
as a localized diagnostic; \KSKSCompileOK{} of \KSTotalExamples{} examples
compile and \KSMakeOK{} build from one source into libbpf-compatible projects;
and the generated objects verifier-load, attach, pass traffic, and run a
tcp-congestion struct\_ops object on a real kernel via a direct libbpf runner.
The result is a concrete, reproducible design point: typing eBPF's
cross-boundary structure is feasible, and because the prototype lowers through
the stock toolchain, both what the type system buys and the version-skew limits
it cannot absorb stay visible and measurable.
